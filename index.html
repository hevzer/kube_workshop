Un peu d'Histoire

  • L'Héritage Google : Né de l'expérience de Google avec ses systèmes d'orchestration internes colossaux : Borg et Omega.
  • L'Open Source (2014) : Google libère le projet publiquement. Le nom "Kubernetes" vient du grec ancien et signifie « Timonier » ou « Pilote ».
  • La Naissance de la CNCF (2015) : Sortie de la v1.0. Kubernetes devient le tout premier projet géré par la Cloud Native Computing Foundation.
  • L'Anecdote du Logo : Les 7 branches de la barre à roue (le logo de K8s) font référence au nom de code original du projet : Project Seven of Nine.

Les Besoins Architecturaux

Quels défis les architectures modernes (Microservices) doivent-elles relever ?

Haute Disponibilité

Tolérance aux pannes

Garantir que l'application reste accessible 24/7, même si un composant matériel (nœud) ou logiciel (conteneur) subit une défaillance critique.

Scalabilité

Adaptation à la charge

Être capable d'absorber instantanément les pics de trafic en multipliant les instances, puis réduire la voilure pour optimiser les coûts Cloud.

Automatisation

Vitesse & Fiabilité

Déployer de nouvelles versions de manière fluide (Rolling Updates) et pouvoir revenir en arrière instantanément (Rollbacks) en cas de bug en production.

Brisons un mythe : Docker vs Kubernetes

Non, Kubernetes ne "remplace" pas Docker. Ils travaillent ensemble.

Docker (L'Ouvrier)

  • Objectif : Créer et empaqueter (Build).
  • Périmètre : Une seule machine hôte.
  • Action : Exécute l'application de manière standardisée et isolée. Il fabrique la brique.

Kubernetes (L'Architecte)

  • Objectif : Gérer et orchestrer.
  • Périmètre : Un cluster entier (des dizaines de serveurs).
  • Action : Décide où et quand déployer les briques Docker pour construire l'immeuble.

L'analogie du Chef d'Orchestre

  • Les Musiciens (Les Conteneurs)

    Chacun exécute une partie de l'application (API, BDD, Frontend).

  • Le Chef d'Orchestre (Kubernetes)

    Coordonne, ajuste le tempo, corrige les fausses notes.

  • La Salle de Concert (Le Cluster)

    L'ensemble des machines hôtes (ressources CPU, RAM).

Pourquoi Kubernetes ?

Sans Orchestrateur (Chaos) :

  • Démarrage manuel de dizaines de conteneurs.
  • Un conteneur plante = Application en panne.
  • Pic de trafic = Serveur saturé (Crash).

Avec Kubernetes (Harmonie) :

  • Création automatisée via des règles.
  • Auto-réparation : Détection et relance automatiques.
  • Autoscaling : Ajout de conteneurs à la volée.

L'Architecture du Cluster

La séparation stricte entre le pilotage (Control Plane) et l'exécution (Worker Nodes).

Le Cerveau vs Les Muscles

Le Control Plane

Le Cerveau

C'est le centre de contrôle global. Il prend les décisions globales concernant le cluster (comme la planification des charges de travail), détecte et réagit aux événements du cluster (comme le démarrage d'un nouveau pod lorsqu'un réplica est manquant).

Les Worker Nodes

Les Muscles

Ce sont les machines (virtuelles ou physiques) qui exécutent réellement vos applications. Chaque nœud est géré par le Control Plane et contient les services nécessaires pour faire tourner les Pods.

Composants du Control Plane

API Server

Le point d'entrée

Le frontend du Control Plane. Il expose l'API Kubernetes. Toute communication, qu'elle vienne de vous (kubectl) ou des composants internes, passe par lui.

etcd

La mémoire centrale

Base de données clé-valeur hautement disponible et cohérente. Elle stocke l'intégralité des données du cluster (configuration, état désiré vs état actuel).

Scheduler

Le répartiteur

Surveille la création des Pods sans nœud assigné et sélectionne le meilleur Worker Node pour les exécuter, en fonction des contraintes de ressources.

Composants des Worker Nodes

Kubelet

Le contremaître

L'agent principal sur chaque nœud. Il communique avec l'API Server et s'assure que les conteneurs décrits dans un PodSpec tournent et sont sains.

Kube-Proxy

L'aiguilleur réseau

Maintient les règles réseau sur le nœud. Ces règles permettent la communication réseau vers les Pods depuis l'intérieur ou l'extérieur du cluster.

Container Runtime

Le moteur

Le logiciel responsable de l'exécution des conteneurs (Containerd, CRI-O). Il télécharge les images et gère le cycle de vie du processus isolé.

Le Langage : Manifestes YAML

apiVersion: v1
kind: Pod
metadata:
  name: mon-pod
spec:
  containers:
  - name: nginx
apiVersion:

Définit la version de l'API Kubernetes à utiliser pour créer cet objet.

kind:

Le type de ressource que l'on souhaite créer (Pod, Service, Deployment...).

metadata:

Données qui aident à identifier l'objet de manière unique (nom, labels, namespace).

spec:

Le cœur du manifeste. Décrit l'état souhaité exact de la ressource (images, ports).

Le Pod : L'unité de base

  • L'atome de K8s : Dans Kubernetes, on ne déploie jamais un conteneur seul, on déploie toujours un Pod.
  • Cohésion : Un Pod encapsule un ou plusieurs conteneurs fortement couplés (ex: une app principale + un conteneur de logs "sidecar").
  • Partage : Les conteneurs d'un même Pod partagent le même espace réseau (une seule IP) et les mêmes volumes de stockage.
  • Mortel : Les Pods sont éphémères. S'ils meurent, on ne les réanime pas, Kubernetes en crée de nouveaux.

Fonctionnalités clés :

  • Disponibilité : Garantit qu'un nombre exact de répliques tourne en permanence (géré via le ReplicaSet).
  • Rolling Updates : Mise à jour progressive d'une nouvelle version d'image sans aucune coupure de service.
  • Rollbacks : Retour en arrière immédiat et simplifié vers une version précédente en cas d'erreur de déploiement.

Exposer une application : Les Services

ClusterIP

Interne par défaut

Expose le Service sur une IP interne au cluster. L'application est uniquement accessible depuis l'intérieur du cluster.

NodePort

Ouverture Statique

Ouvre un port spécifique (entre 30000 et 32767) sur chaque nœud du cluster. Permet d'accéder à l'application depuis l'extérieur.

LoadBalancer

Trafic Externe

Demande automatiquement la création d'un équilibreur de charge au fournisseur Cloud sous-jacent (AWS, Azure, GCP).

Routage Avancé : L'Ingress

Le point d'entrée HTTP/S

L'Ingress gère les accès externes aux services dans un cluster en agissant comme un reverse proxy (ex: Nginx, Traefik).

  • Routage intelligent : Dirige le trafic vers différents services selon le chemin (ex: /api vs /web).
  • Virtual Hosting : Permet d'héberger plusieurs applications/domaines sur une seule adresse IP publique.
  • Sécurité : Gère la terminaison SSL/TLS (les certificats HTTPS) de manière centralisée.

Persistance du Stockage

Les Pods sont éphémères, comment sauvegarder les données (Base de données) ?

Persistent Volume (PV)

La Ressource Physique : C'est un morceau de stockage dans le cluster (NFS, AWS EBS, Disque local) qui a été provisionné par un administrateur. Son cycle de vie est indépendant de tout Pod.

Persistent Volume Claim (PVC)

La Demande (Le ticket) : C'est une requête de stockage faite par un utilisateur/Pod (ex: "J'ai besoin de 5Go"). Kubernetes va chercher un PV libre qui correspond à cette demande et les lier (Bind).

Configuration & Isolation

Namespaces

Les Murs

Des clusters virtuels pour isoler et organiser vos ressources au sein d'un même cluster physique (ex: dev, staging, prod).

ConfigMaps

La Configuration

Externalise les variables d'environnement et les fichiers de configuration non sensibles hors de l'image Docker.

Secrets

Données Sensibles

Stocke de manière plus sécurisée les mots de passe, clés d'API et certificats (Les données sont encodées en Base64).

L'Écosystème : Helm

  • Le Package Manager : Helm est à Kubernetes ce que apt est à Ubuntu ou npm à Node.js.
  • Les Charts : Il regroupe vos fichiers YAML (Deployments, Services, Secrets) dans un format packagé appelé "Chart".
  • Templates : Il permet de rendre vos YAML dynamiques avec des variables (ex: changer le port ou le tag d'image selon l'environnement).
  • Déploiement Simplifié : Installez des applications complexes (comme Prometheus ou une base PostgreSQL complète) en une seule commande : helm install bitnami/postgresql

L'Environnement Local : Minikube

  • Qu'est-ce que c'est ?

    Un outil qui implémente un cluster Kubernetes local (sur un seul nœud) directement sur votre machine portable.

  • Le but ?

    Idéal pour l'apprentissage, les tests et le développement quotidien (Proof of Concept) avant de déployer dans le Cloud.

  • L'Outil de Commande

    Nous utiliserons kubectl pour interagir avec l'API de notre Minikube.

Les alternatives : K3s & MicroK8s

D'autres distributions légères pour le développement local et l'Edge computing.

K3s

Par Rancher (SUSE)

Une distribution Kubernetes certifiée et ultra-légère. Emballée dans un binaire unique de moins de 100 Mo. Parfaitement adaptée pour l'embarqué.

🎯 Cibles : IoT, Edge computing, architectures ARM, et intégration CI/CD.

MicroK8s

Par Canonical

L'approche "Zero-Ops" de Kubernetes. S'installe en une seule commande via les paquets Snap. Très apprécié pour sa modularité grâce aux add-ons natifs.

🎯 Cibles : Développeurs en local, postes de travail Linux, appliances isolées, tests rapides.

Le Kit de Survie : Troubleshooting

  • 1. Inspecter

    Affiche les événements système (Events) qui expliquent pourquoi K8s refuse de lancer le pod (ex: Image introuvable).

  • 2. Lire les logs

    Affiche la sortie standard de votre conteneur (les erreurs de votre code). Ajoutez -f pour le direct.

  • 3. Entrer dedans

    Ouvre un terminal interactif à l'intérieur du conteneur pour tester le réseau ou vérifier un fichier.

bash - kubectl
# 1. Inspecter l'état et les events $kubectl describe pod <nom-du-pod>
# 2. Afficher les logs applicatifs $kubectl logs <nom-du-pod>
# 3. Ouvrir un shell dans le conteneur $kubectl exec -it <nom-du-pod> -- /bin/sh

Travaux Pratiques (TP)

Ouvrez vos terminaux. Il est temps de mettre les mains dans le cambouis.

TP 1 (1/2) : Démarrage & Création

  • 1. Démarrage local

    Démarrez Minikube. Il va télécharger l'image K8s et configurer automatiquement kubectl pour vous.

  • 2. Lancement du Pod

    Lancez un Pod Nginx en mode impératif (commande directe) sans utiliser de fichier YAML pour l'instant.

bash - minikube
# 1. Démarrer le cluster $minikube start
# 2. Créer un pod simple $kubectl run mon-premier-pod --image=nginx:alpine

TP 1 (2/2) : Observation & Nettoyage

  • 3. Observation

    Vérifiez le statut global, puis analysez les "Events" en détail pour voir les étapes de création.

  • 4. Nettoyage

    Supprimez le pod. Observez qu'il ne réapparaît pas (car il n'y a pas de Deployment pour le protéger).

bash - minikube
# 3. Lister et inspecter $kubectl get pods
$kubectl describe pod mon-premier-pod
# 4. Supprimer le pod $kubectl delete pod mon-premier-pod

TP 2 (1/2) : Workload & Scalabilité

  • 1. Déploiement

    Au lieu d'un Pod isolé, créez un contrôleur résilient qui maintiendra votre application en vie.

  • 2. Scalabilité

    Augmentez le nombre de réplicas à 3 et vérifiez la création instantanée des nouveaux Pods.

bash - minikube
# 1. Créer le Deployment $kubectl create deployment web-app --image=nginx:alpine
# 2. Mettre à l'échelle (Scale) $kubectl scale deployment web-app --replicas=3
$kubectl get pods

TP 2 (2/2) : Exposition réseau

  • 3. Service NodePort

    Rendez l'application accessible depuis l'extérieur du cluster via un port généré aléatoirement.

  • 4. Accès HTTP

    Utilisez l'utilitaire de Minikube pour créer le tunnel réseau et ouvrir le site dans votre navigateur.

bash - minikube
# 3. Exposer le port cible $kubectl expose deployment web-app --type=NodePort --port=80
# 4. Ouvrir le site web $minikube service web-app

TP 3 (1/2) : Interface Headlamp

  • 1. YAML distant

    Déployez ce Dashboard d'administration directement depuis son dépôt officiel sur GitHub.

  • 2. Port-Forward

    Créez un tunnel temporaire (qui bloquera ce terminal) entre votre machine et le cluster.

bash - minikube
# 1. Appliquer un manifeste distant $kubectl apply -f https://raw.githubusercontent.com/headlamp-k8s/headlamp/main/kubernetes-headlamp.yaml
# 2. Créer le tunnel vers le Service $kubectl port-forward svc/headlamp 4000:80 -n kube-system

TP 3 (2/2) : Accès à l'Interface

  • 3. Le Token

    Ouvrez un nouveau terminal et générez un jeton administrateur pour vous connecter.

  • 4. Visualisation

    Ouvrez votre navigateur, collez le token, et admirez visuellement vos ressources !

bash - minikube (Nouveau Tab)
# 3. Générer le jeton de sécurité $kubectl create token default
# 4. URL à ouvrir dans le navigateur >http://localhost:4000
